# DEMO PRD：基于靶点情报的 药物研发立项决策支持子智能体 Sub-Agent

- **版本**：v0.1
- **目标**：一个可以快速搭建的 DEMO

## 1. DEMO 背景

药物研发早期立项时，团队通常需要快速回答几个关键问题：

- 这个靶点有没有足够的科学依据？
- 有没有公开文献支持？
- 有没有相关临床试验信号？
- 竞争是否已经很拥挤？
- 是否存在明显的风险或负面证据？
- 从公开信息看，是否值得继续投入做深入评估？

这些信息分散在多个公开来源中，例如：

- **PubMed**
- [ClinicalTrials.gov](http://ClinicalTrials.gov)
- **Google Scholar**
- 企业内部资料（DEMO可以略去。或者模拟1-2份内部资料）

传统做法依赖人工检索、人工筛选、人工总结，效率低、容易漏信息，也很难快速形成一个可复核的初步建议。

因此，本 DEMO 希望做一个**Agent based 研究助手原型**，用微软工具 + 开源工具 + 免费公开数据源，快速完成一个“靶点立项初筛”的决策支持流程。

## 2. DEMO 目标

### 2.1 总体目标

实现一个可运行的 DEMO，使用户输入一个 **靶点名称**（可附带疾病/适应症），系统自动完成：

1. 检索公开科学文献
2. 检索临床试验信息
3. 检索 Google Scholar 结果作为补充
4. 检索企业内部资料库
5. 汇总证据并生成结构化结论
6. 输出一个带引用的 **Go / No-Go / Need More Data** 建议，并对引用的证据附带原文链接

### 2.2 DEMO 不追求的内容

本期 DEMO **不做**：

- 全 10 个 sub-agent 全量实现
- 企业级权限体系
- 商业数据库接入
- 全自动高精度专利分析
- 完整财务建模
- 真正替代研发委员会决策等等…

这个 DEMO 只是一个 **“立项早筛智能助手”原型**。

## 3. DEMO 核心思路

### 3.1 Agent 架构

采用一个简化版架构：

#### 1）Orchestrator Agent

作为核心调度和总结 Agent，负责：

- 接收用户问题
- 拆分研究任务
- 调用各 sub-agent
- 汇总各 sub-agent 输出
- 形成最终建议：
  - **Go**
  - **No-Go**
  - **Need More Data**

#### 2）Sub Agent（DEMO 版建议只做 3~4 个）

为了控制复杂度，本 DEMO 建议先实现以下 4 个：

#### A. 文献研究 Agent

负责：

- 查询 PubMed
- 查询 Google Scholar
- 提取与靶点相关的核心研究结论
- 给出文献支持强度判断
- 保留引用链接、标题、摘要摘要化结果

#### B. 临床试验情报 Agent

负责：

- 查询 [ClinicalTrials.gov](http://ClinicalTrials.gov)
- 找到与靶点、机制、适应症相关的临床试验
- 提取试验阶段、状态、适应症、干预方式等信息
- 识别积极信号和失败信号

#### C. 竞争与情报 Agent

负责：

- 基于公开网页 / Scholar / PubMed / [ClinicalTrials.gov](http://ClinicalTrials.gov) 的补充信息
- 判断是否已有大量竞争者
- 判断该靶点是否已经过热、过卷、过晚进入

#### D. 决策总结 Agent（可并入 Orchestrator）

负责：

- 整合前三个 Agent 结果
- 输出结构化评分或结论
- 说明为什么给出 Go / No-Go / Need More Data

## 4. 用户场景

### 场景 1：靶点初筛

用户输入：

- 靶点：GLP-1R
- 适应症：肥胖 / 2 型糖尿病

系统输出：

- PubMed 研究概况
- Google Scholar 补充证据
- [ClinicalTrials.gov](http://ClinicalTrials.gov) 相关试验概况
- 初步建议：Go / No-Go / Need More Data

### 场景 2：靶点 + 疾病联合评估

用户输入：

- 靶点：TL1A
- 适应症：炎症性肠病

系统输出：

- 靶点生物学支持
- 当前临床进度
- 竞争拥挤程度
- 是否值得继续跟进

## 5. 功能需求

### 5.1 输入

用户在前端输入以下内容：

#### 必填项

- 靶点名称（例如：GLP-1R、TL1A、PCSK9）

#### 可选项

- 疾病/适应症
- 同义词/别名
- 研究重点（如：看临床信号 / 看基础研究 / 看竞争格局）
- 时间范围（过去 5 年、10 年等）

### 5.2 数据源

本 DEMO 使用 **免费公开数据源**：

#### 必选

1. **PubMed**
   * 用于科学文献检索
   * 提供标题、摘要、PMID、链接等
2. [ClinicalTrials.gov](http://ClinicalTrials.gov)
   * 用于临床试验信息检索
   * 提供试验阶段、状态、适应症、干预等信息
3. **Google Scholar**
   * 用于补充学术搜索广度
   * 弥补 PubMed 以外的研究线索和综述线索

#### 实现分层建议

- **权威结构化证据层**：PubMed、[ClinicalTrials.gov](http://ClinicalTrials.gov) 优先走官方 API
- **开放网页补充层**：Google Scholar 相关检索优先通过搜索代理、第三方搜索 API 或宿主环境提供的 web search 能力实现，不建议 DEMO 阶段直接做 Scholar 爬虫
- **LLM 综合层**：使用 Microsoft AI Foundry 中的 GPT / Claude 模型做任务规划、证据归纳、结论生成

#### 实现约束

- 不建议把所有搜索都交给通用 web search
- 医学主证据必须优先保留官方来源链接与结构化字段
- 通用搜索主要用于补充综述、公开网页、竞争情报和 Scholar 线索
- 如果运行环境不提供可控的 web search，则默认回退到第三方搜索 API 适配器

#### 可选补充 (不建议这次做)

1. **OpenAlex**
   * 免费开放学术元数据
   * 可作为 Scholar 的补充或替代索引
2. **Europe PMC**
   * 作为医学文献补充来源
3. **公开网页 / 公司新闻稿**
   * 用于补充竞争与管线情报

### 5.3 文献研究 Agent 功能

#### 输入

- 靶点名称
- 疾病/适应症（可选）

#### 输出

- 相关论文列表
- 高相关摘要总结
- 文献支持强度判断
- 正面证据 / 负面证据
- 引用列表

#### 最低实现要求

- 支持 PubMed 检索
- 支持 Google Scholar 相关补充检索
- 取前 N 条结果（例如 10~20 条）
- 用 LLM 生成简明综述
- 输出引用链接

#### 工程建议

- PubMed：优先用官方 API / E-utilities
- Google Scholar：优先通过搜索代理、第三方搜索 API 或 web search 能力实现“摘要级补充检索”，不建议 DEMO 阶段直接抓取页面
- 不要一开始就做复杂全文解析，先做标题 + 摘要 + 元数据汇总
- 对 Scholar / Web 结果要保留原始标题、来源站点、跳转链接，避免只保留 LLM 摘要

### 5.4 临床试验情报 Agent 功能

#### 输入

- 靶点名称
- 疾病/适应症（可选）

#### 输出

- 相关临床试验列表
- 阶段分布（Phase 1/2/3）
- 试验状态（Recruiting / Completed / Terminated 等）
- 潜在积极信号
- 潜在失败/风险信号

#### 最低实现要求

- 支持 [ClinicalTrials.gov](http://ClinicalTrials.gov) 检索
- 提取试验标题、NCT 号、阶段、状态、适应症、干预
- 形成一个简明表格或结构化 JSON
- LLM 总结趋势

### 5.5 竞争与情报 Agent 功能

#### 输入

- 靶点名称
- 疾病/适应症（可选）

#### 输出

- 竞争活跃程度
- 主要研究热点
- 是否存在拥挤赛道迹象
- 是否能找到公司/管线公开动态

#### 最低实现要求

- 基于 PubMed / Scholar / [ClinicalTrials.gov](http://ClinicalTrials.gov) 的结果二次归纳
- 支持公开网页标题与摘要片段补充
- 不要求 DEMO 阶段做完整专利数据库接入

### 5.6 Orchestrator Agent 功能

#### 输入

- 用户研究任务

#### 处理逻辑

1. 解析用户问题
2. 客户确认解析是正确的，可以Say No或者手动输入改正信息（体现Human in the loop）
3. 生成子任务
4. 客户确认子任务是正确的，可以Say No或者手动输入改正信息，重新生成子任务。
5. 调用文献研究 Agent
6. 调用临床试验情报 Agent
7. 调用竞争与情报 Agent
8. 汇总结果
9. 输出决策建议

#### 输出格式

建议固定成下面结构：

- **目标靶点**
- **适应症**
- **文献支持总结**
- **临床试验总结**
- **竞争格局总结**
- **主要风险**
- **主要机会**
- **结论建议**
  - Go / No-Go / Need More Data
- **引用列表**

## 6. 决策逻辑（DEMO 版）

DEMO 不建议做太复杂的数学模型，可以先做 **规则 + LLM 总结** 的混合方式。

#### 示例规则

如果满足以下条件，可偏向 **Go**：

- PubMed 中存在较多高相关研究
- 有明确正向生物学机制支持
- [ClinicalTrials.gov](http://ClinicalTrials.gov) 中已有积极临床信号
- 竞争未完全饱和

如果满足以下条件，可偏向 **No-Go**：

- 文献支持弱
- 缺少清晰机制证据
- 相关临床多失败/终止
- 竞争极度拥挤且差异化不明显

如果证据不足，则输出：

- **Need More Data**

#### 要求

最终结论必须给出：

- 原因说明
- 证据来源
- 不确定性说明

## 7. 技术方案建议

### 7.1 微软工具优先方案

优先使用以下微软工具：

#### 模型层

- **建议 Microsoft AI Foundry 选择 GPT 或 Claude 模型**
  - 用于任务规划、摘要、综合分析、结论生成

#### 前端

- **自建轻量 Web 页面**
  - 用于输入靶点、展示结构化结果、展示引用和导出报告
  - 首版优先考虑单页应用或后端模板页，降低联调复杂度

#### 编排层

- **Python 编排服务 + Foundry 模型调用**
  - Orchestrator 负责任务解析、用户确认、调用各子 Agent、聚合结论
  - 子 Agent 在实现上可以是同一服务内的模块，而不是多个独立部署单元

#### 部署运行时

- **Azure Container Apps**
  - 承载 Python 编排服务和轻量 Web 页面
- **Azure Container Registry**
  - 保存应用镜像，供 Container Apps 拉取

#### 检索与缓存

- **Azure AI Search**（可选）
  - 如果要做索引缓存和二次检索可加入
  - DEMO 阶段不是必须

#### 搜索接入建议

- **PubMed API / E-utilities**：主文献证据来源
- **ClinicalTrials.gov API**：主临床试验证据来源
- **Web Search / 第三方搜索 API**：Scholar 补充线索、公开网页、竞争情报来源

#### 存储

- **Azure Blob Storage**
  - 保存查询结果、导出报告
- **Azure Cosmos DB for NoSQL**
  - 保存任务记录、执行状态、结构化结果和引用清单

#### 可观测性

- **Azure Application Insights + Log Analytics**
  - 记录每次检索耗时、模型调用、错误、部分失败和导出状态

## 7. 系统流程

### Step 1：用户输入

用户输入：

- 靶点
- 适应症（可选）
- 查询目标

### Step 2：Orchestrator 和用户确认问题理解和解析是否正确

### Step 3：Orchestrator agent开始规划一下步骤

例如：

- 去 PubMed 找科学支持
- 去 Google Scholar 找补充学术证据
- 去 [ClinicalTrials.gov](http://ClinicalTrials.gov) 找临床信号
- 汇总竞争与风险

### Step 4：各 Sub Agent 分别执行

每个 Agent 返回结构化 JSON：

- 来源
- 来源类型（PubMed / ClinicalTrials / Web）
- 摘要
- 关键发现
- 证据链接
- 结构化字段
- 置信度 / 风险提示

### Step 5：Orchestrator 汇总

生成一份最终报告。

### Step 6：前端展示

展示：

- 总结结论
- 关键证据
- 引用列表
- 决策建议

## 9. 前端展示建议

DEMO 前端不要复杂，建议只做一个简单页面：

#### 界面语言

- **采用中文界面**，所有标签、按钮、提示信息均为中文

#### 页面布局

- **单页滚动布局**：输入区、确认区、运行状态、结果区在同一页面内纵向排列
- 输入区始终可见在顶部（运行时禁用输入）
- 确认面板在输入区下方展开（而非替换输入区）
- 运行进度条在确认面板下方
- 结果区在最下方，完成后自动滚动到结果区
- 不做页面跳转

#### 输入区

- 靶点名称
- 适应症
- 同义词 / 别名（可选）
- 研究重点（可选：全部领域 / 文献 / 临床 / 竞争）
- 时间范围（可选：近 3/5/10 年）
- 查询按钮

#### 输出区

#### 1）总体建议卡片

- Go / No-Go / Need More Data

#### 2）文献支持

- 3~5 条核心摘要
- 关键文献引用

#### 3）临床试验情报

- 试验表格
- 阶段 / 状态统计

#### 4）竞争与风险

- 主要竞争点
- 主要风险点

#### 5）引用区

- PubMed 链接
- [ClinicalTrials.gov](http://ClinicalTrials.gov) 链接
- Google Scholar / Web 搜索结果链接

## 10. 输出物

DEMO 至少输出两种形式：

### 10.1 页面结果

前端页面展示结果

### 10.2 导出报告

导出 Markdown / Word / PDF 报告，内容包括：

- 查询目标
- 摘要结论
- 证据清单
- 风险与机会
- 最终建议

## 11. 非功能需求

### 11.1 可解释性

- 每条关键结论要有来源
- 不能只给答案不给证据

### 11.2 成本控制

- 只抓前5条结果
- 避免无限 Deep Research
- 先做轻量总结

### 11.3 稳定性

- 某个数据源失败时，系统要提示而不是崩掉
- 要允许部分结果返回

## 12. MVP 范围

建议工程师先做最小闭环：

#### MVP 必做

- 输入靶点
- PubMed 检索
- [ClinicalTrials.gov](http://ClinicalTrials.gov) 检索
- Google Scholar / Web 搜索补充检索
- LLM 汇总
- 输出 Go / No-Go / Need More Data
- 带引用链接

#### MVP 不做

- 登录系统
- 多用户权限
- 长期知识库
- 专利数据库接入
- 复杂图谱分析
- 多轮会话记忆

## 13. 成功标准

如果 DEMO 能做到以下几点，就算成功：

1. 用户输入一个靶点后，10 分钟内得到结果
2. 结果里有明确引用来源
3. 能看出文献支持与临床信号
4. 能给出初步 Go / No-Go / Need More Data 建议
5. 工程师可以在 1~2 周内做出第一版可演示原型

## 14. 实现建议

为了提高落地速度，建议工程实现分三步：

### 第一阶段：打通链路

- 接 PubMed
- 接 [ClinicalTrials.gov](http://ClinicalTrials.gov)
- 做一个 Google Scholar / Web 搜索补充检索适配层
- 把结果喂给 LLM 做总结

### 第二阶段：结构化输出

- 每个 Agent 输出统一 JSON
- 前端做结果展示
- 增加引用列表和链接

### 第三阶段：导出报告

- 把结果导出成 Markdown / Word
- 形成可演示成果

## 16. 一句话总结

这个 DEMO 不是为了“证明 AI 什么都能做”，而是为了证明：

微软工具 + 免费公开数据源，能够快速搭一个药物研发立项决策支持 Agent 原型，帮助团队对某个靶点做初步 Go / No-Go 判断。
